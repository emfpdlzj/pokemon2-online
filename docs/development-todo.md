/# Development TODO

현재 코드 기준으로, 실제 제품화에 필요하지만 아직 남아 있는 핵심 개발 항목을 정리한다.

## 현재 우선순위

### 1. 사용자 식별과 세이브 소유권 도입

- 세이브 데이터에 사용자 식별자 필드를 추가한다.
- `mode + slotNumber` 기준이 아니라 `user + mode + slotNumber` 기준으로 저장 구조를 바꾼다.
- 세이브 조회, 수정, 삭제 시 소유자 검증을 붙인다.
- 멀티플레이 입장과 재접속에도 사용자 식별 정책을 연결한다.

완료 기준:

- 다른 사용자가 남의 세이브를 조회하거나 덮어쓸 수 없다.
- 동일 사용자는 자신의 슬롯 1~3만 안정적으로 관리할 수 있다.

### 2. LLM 호출을 서버로 이관

- 브라우저에서 직접 LLM API 키를 쓰지 않도록 구조를 바꾼다.
- 서버에서 프롬프트 생성, 모델 호출, 에러 처리, 응답 검증을 담당한다.
- 호출 제한, 실패 로그, 비용 추적이 가능하도록 설계한다.

완료 기준:

- 클라이언트 번들 및 런타임 설정에 비밀 키가 노출되지 않는다.
- LLM 실패 시 사용자에게 일관된 fallback 응답을 제공한다.

### 3. 멀티플레이 연결 안정성 보강

- WebSocket 종료 시 재접속 또는 재입장 흐름을 정의한다.
- `player_joined`, `player_left`, `chat` 이벤트를 클라이언트 UI에 반영한다.
- 연결 끊김, 서버 거부, 방 만석, 재입장 실패 상태를 사용자에게 명확히 보여준다.

완료 기준:

- 접속 종료 후 사용자가 상태를 이해하고 다시 진입할 수 있다.
- 방 안의 다른 플레이어 변화가 화면과 UI에 반영된다.

### 4. 멀티 모드 세이브 UI와 운영 흐름 정리

- 서버는 `single`, `multi` 모드 세이브를 지원하지만 클라이언트 세이브 메뉴는 현재 `single`만 조회한다.
- 멀티 캐릭터 저장/불러오기 진입점, 슬롯 표시 방식, 싱글 저장과의 구분 UI를 정의한다.
- 멀티 모드 저장이 방 진입/이탈 흐름과 어떻게 연결되는지 명확히 한다.

완료 기준:

- 사용자가 UI에서 싱글 저장과 멀티 저장을 구분해서 관리할 수 있다.
- 멀티플레이 진행 상태를 어떤 시점에 어떤 슬롯에 저장하는지 설명 가능하다.

### 5. 세이브 동기화 신뢰성 보강

- 서버 저장 실패를 조용히 무시하지 않고 사용자에게 알려준다.
- 재시도 정책, 로컬 fallback 상태, 마지막 서버 저장 시점을 표시한다.
- 서버 세이브와 로컬 세이브가 충돌할 때의 우선순위를 정한다.

완료 기준:

- 저장 성공/실패를 사용자가 확인할 수 있다.
- 저장 충돌이나 오프라인 상황에서도 데이터 유실 가능성을 설명할 수 있다.

### 6. 운영 보안과 배포 하드닝

- 공개 admin metrics 엔드포인트 보호 정책을 추가한다.
- CORS, 환경변수, DB 연결 정책을 운영 환경 기준으로 분리한다.
- `EnsureCreated` 중심 초기화 대신 migration 기반 DB 관리로 전환한다.
- 기본 PR CI에서 클라이언트/서버 테스트를 자동 실행하도록 구성한다. (`2026-06-15` 완료)

완료 기준:

- 운영 환경에서 공개되면 안 되는 API가 보호된다.
- 스키마 변경 이력이 migration으로 관리된다.
- 기본 테스트가 수동 실행에 의존하지 않는다.

### 7. 대외 소개 문구와 실제 구현 정합성 정리

- 포트폴리오/소개 문서에서 현재 저장소와 맞지 않는 기술 스택이나 기능 설명을 제거한다.
- `Kotlin`, `FastAPI`, `UnityAR`, `STT`처럼 현재 저장소에 없는 기술 표기를 정리한다.
- 채팅 동기화처럼 서버 프로토콜은 있지만 클라이언트 UX가 비어 있는 항목은 구현 범위에 맞게 표현을 조정한다.

완료 기준:

- README, 포트폴리오 소개문, 외부 제출 문구가 현재 저장소 구현과 일치한다.
- 검증되지 않은 기술 또는 미완 기능이 완료된 것처럼 소개되지 않는다.
