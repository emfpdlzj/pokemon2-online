/# Development TODO

현재 코드 기준으로, 실제 제품화에 필요하지만 아직 남아 있는 핵심 개발 항목을 정리한다.

## 현재 우선순위

### 1. 멀티 모드 세이브 UI와 사용자 이전 정책 정리

- 서버는 `single`, `multi` 모드 세이브를 지원하지만 클라이언트 세이브 메뉴는 여전히 `single`만 조회한다.
- 멀티 캐릭터 저장/불러오기 진입점, 슬롯 표시 방식, 싱글 저장과의 구분 UI를 정의한다.
- 멀티 모드 저장이 방 진입/이탈 흐름과 어떻게 연결되는지 명확히 한다.
- 사용자 식별 도입 전의 공용 세이브를 어떤 사용자에게 귀속시킬지 운영 전환 정책을 정리한다.

완료 기준:

- 사용자가 UI에서 싱글 저장과 멀티 저장을 구분해서 관리할 수 있다.
- 멀티플레이 진행 상태를 어떤 시점에 어떤 슬롯에 저장하는지 설명 가능하다.
- 기존 세이브를 새 사용자 구조로 옮길 때 데이터 소유권 기준이 정해져 있다.

### 2. 세이브 동기화 신뢰성 보강

- 서버 저장 실패를 조용히 무시하지 않고 사용자에게 알려준다.
- 재시도 정책, 로컬 fallback 상태, 마지막 서버 저장 시점을 표시한다.
- 서버 세이브와 로컬 세이브가 충돌할 때의 우선순위를 정한다.

완료 기준:

- 저장 성공/실패를 사용자가 확인할 수 있다.
- 저장 충돌이나 오프라인 상황에서도 데이터 유실 가능성을 설명할 수 있다.

### 3. 운영 보안과 배포 하드닝

- 공개 admin metrics 엔드포인트 보호 정책을 추가한다.
- `EnsureCreated` 중심 초기화 대신 migration 기반 DB 관리로 전환한다.
- 사용자 식별 도입 이후 부하 테스트와 운영 도구가 새 인증/접속 흐름을 따라가도록 갱신한다.
- 운영 배포 시 필요한 비밀값, allowlist, DB 스키마 적용 순서를 문서화한다.

완료 기준:

- 운영 환경에서 공개되면 안 되는 API가 보호된다.
- 스키마 변경 이력이 migration으로 관리된다.
- 운영 검증용 스크립트와 문서가 현재 인증/배포 흐름과 어긋나지 않는다.
